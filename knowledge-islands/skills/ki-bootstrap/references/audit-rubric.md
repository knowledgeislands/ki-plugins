# Audit Rubric — ki-bootstrap

Line-by-line criteria for auditing a Knowledge Islands repo's project-local skill and agent install against [the bootstrap standard](bootstrap-standard.md). Each is tagged **[M] mechanical** (the bundled checker — [`copy-skills.ts`](../scripts/copy-skills.ts) or [`link-agents.ts`](../scripts/link-agents.ts) `--check` decides it — capture its output, don't re-derive) or **[J] judgment** (assess by reading). Severity uses the unified ladder defined in `ki-engineering`'s [`enforcement-framework.md`](../../../foundations/ki-engineering/references/enforcement-framework.md) §2.

## BOOT — project-local skill install

- **BOOT-1 [M]** WARN / FAIL — for **each** runtime the repo declares (`.ki-config.toml` `[ki-repo] target_runtimes`, default `["claude-code"]`), that runtime's project-local skills dir (`.claude/skills/` for Claude Code, `.agents/skills/` for Codex) mirrors the repo's declared coverage as complete generated regular-file copies. Missing, extra, symlinked, or source-drifted payloads are WARN. CONFORM re-copies a payload only when its generated marker (logical source plus deterministic tree digest) still validates; an altered or unfamiliar payload is preserved with a migration diagnostic rather than overwritten. Bare and simply quoted TOML table keys are equivalent; multiline-string content never counts as a header. The harness is checked the same way — it copies its own coverage, not the whole fleet (ADR-KI-HARNESS-007). One case is **FAIL**: a declared or noncanonical ki-like root that resolves to no skill in the ref-specific harness index — almost always a table left behind by an upstream rename or removal (e.g. `ki-websites-11ty` → `ki-website`). It is not auto-renamed (the mapping isn't mechanical), so the `.ki-config.toml` table must be reconciled by hand. Check, write, and dry-run publisher modes all perform this validation before mutation; duplicate roots are diagnosed once in sorted order.
- **BOOT-3 [M]** WARN — each declared runtime's project-local skills dir (`.claude/skills/`, `.agents/skills/`) is gitignored (the copies are generated, never committed). CONFORM writes this line automatically when it copies, per runtime — the write path appends the checker-recognised `<dir>/` form (never a leading-slash variant that would leave the check unsatisfied), creating `.gitignore` if absent.
- **BOOT-4 [J]** — the repo's _declared_ coverage is itself correct — it opts into the skills it actually uses. This is `ki-repo`'s coverage cascade (detected-artifact ⟺ declared-table), not this skill's; route a wrong declaration there rather than papering over it by hand-linking. The keystone (`ki-bootstrap`) must be installed globally for the keystone linker to resolve — an environment precondition, not a repo property.

## BOOT — project-local governance agent install

- **BOOT-6 [M]** WARN — `.claude/agents/` mirrors `agents/governance/*.md` when the repo's `.ki-config.toml` carries the bare `[ki-agents]` table, and is empty when it doesn't — no missing links, no links outside that set, no dangling links. Unlike skills there is no baseline: absent the table, the expected set is empty.
- **BOOT-8 [M]** WARN — `.claude/agents/` is gitignored (the links are generated, never committed). As with BOOT-3, CONFORM writes this line automatically when it links agents (only when the `[ki-agents]` set is non-empty).

## BOOT — vendored self-sufficiency install

- **BOOT-9 [M]** WARN / FAIL — the target's `.ki-meta/skills/` mirrors the harness-side expected set — declared roots ∪ the transitive `implies:` closure, restricted to skills carrying a discoverable checker; there is no injected baseline — with no missing and no extra vendored skill directories. Checked by [`audit.ts`](../scripts/audit.ts), run from the harness (the `implies:` graph lives in source SKILL.md frontmatter, not in anything copied into the target, so this cannot be checked by the target's own standalone `.ki-meta/bin/ki-audit`). Missing or extra vendor directories are WARN and conformable by re-running `bun skills/keystone/ki-bootstrap/scripts/bootstrap.ts <target>`. An exact or dotted declaration with no canonical skill, or an unresolvable explicit EDUCATE seed/dependency, is FAIL: audit emits a structured BOOT-9 finding and EDUCATE exits before `.ki-meta/` mutation. Reconcile names by hand; bootstrap never guesses a rename.
- **BOOT-11 [M]** FAIL / NA — when the target carries matching canonical skill sources in its own `skills/` tree, every direct file-kind audit and conform unit under `.ki-meta/skills/<skill>/` is a regular file that matches its regular-file `skills/<cluster>/<skill>/scripts/<mode>.ts` source byte-for-byte. A missing, symlinked, non-regular, or different source/vendor unit is a FAIL fixed by restoring the canonical source and re-running bootstrap before commit. The harness pre-commit gate runs this audit over a materialized snapshot of the Git index, so partially staged or unstaged working-tree bytes cannot mask commit drift. Generated command wrappers and HELP snapshots are excluded because they are render products rather than direct source copies. A bootstrapped-only target has no canonical sources in its own tree, so this comparison reports NA instead of comparing against whichever harness checkout happens to invoke the audit; its manifest remains the record of the vendored output it received.

## BOOT — aggregate judgmental sweep

- **BOOT-10 [J]** ADVISORY ([ADR-KI-HARNESS-SKILLS-001](../../../../docs/decisions/ADR-KI-HARNESS-SKILLS-001-canonical-modes.md)'s AUDIT/CONFORM contract — "capture the checker's output verbatim, then apply judgment criteria" — applied transitively across every skill the aggregate reports on, closing the loop [ADR-KI-HARNESS-SKILLS-010](../../../../docs/decisions/ADR-KI-HARNESS-SKILLS-010-comparable-cited-checker-findings.md) opened by giving ADVISORY reminders a named consumer) — whenever `.ki-meta/skills/` is non-empty, AUDIT and CONFORM apply each governed skill's own `[J]` judgment criteria in addition to running the mechanical aggregate (`ki:audit`/`ki:conform`) — delegating to the matching `agents/governance/<skill>-lead` agent where one exists, reading that skill's `audit-rubric.md` directly where none does. This is always-on and cannot be mechanically decided (a reminder, never FAIL/WARN); the checker emits it as an ADVISORY finding so the aggregate surfaces it alongside every skill's own findings without re-implementing per-skill judgment here.

## Reporting

Produce findings on the severity ladder, each `severity · criterion · what · fix`. Most BOOT findings are WARN (conformable, never ship-blocking): a missing or drifted skill payload, a missing/dangling agent link, or an absent gitignore entry is fixed by Mode CONFORM (re-run the relevant copier or linker; it recreates the generated surface and appends the gitignore line). Wiring `package.json` convenience keys is out of scope here — that is `ki-engineering`'s concern. The exceptions are BOOT-1 and BOOT-9 unresolvable-root cases, where CONFORM cannot infer a rename, and BOOT-11 source-copy drift, which must block a harness commit until re-bootstrap restores the exact copy. Close by naming the composition: `ki-repo` owns whether declared coverage is right and owns the file-level/foundation-scaffold contract; each sibling skill retains its own table schema and conformer. Whether a repo should opt into `[ki-agents]` at all is likewise a judgment call for the repo owner, not this skill.
